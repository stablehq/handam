# E2E 테스트 리뷰: 기존 분석 + 개선 방안

> 기존 코드(`e2e_test.py`, `e2e_test_part2.py`)와 테스트 계획서(`e2e-test-plan.md`)를 실제 코드와 교차검증한 결과

## 1. 기존 E2E의 핵심 문제: "가짜 PASS"가 너무 많다

현재 E2E의 가장 큰 문제는 **실제로 API를 호출하지 않고 PASS를 찍는 테스트가 절반 이상**이라는 것.

### 가짜 PASS 목록 (코드 리뷰만으로 PASS 처리된 항목)

| 섹션 | 가짜 PASS | 실제 API 호출 PASS | 비율 |
|------|----------|------------------|------|
| C (객실 자동배정) | C-2~C-10 (9건) | C-1 (1건) | 90% 가짜 |
| F (target_mode) | F-2,F-4,F-6,F-8,F-9 (5건) | F-1,F-3,F-5,F-7 (4건) | 56% 가짜 |
| H (구조적 필터) | H-6~H-13 (8건) | H-1~H-5 (5건) | 62% 가짜 |
| I (stay_filter) | I-3~I-6 (4건) | I-1,I-2 (2건) | 67% 가짜 |
| J (send_condition) | J-2~J-6 (5건) | J-1 (1건) | 83% 가짜 |
| K (event) | K-2~K-6 (5건) | K-1 (1건) | 83% 가짜 |
| L (칩) | L-4~L-12 (9건) | L-1~L-3 (3건) | 75% 가짜 |
| N (발송) | N-2~N-10 (9건) | N-1 (1건) | 90% 가짜 |
| Q (멀티테넌트) | Q-11~Q-14 (4건) | Q-1~Q-10 (10건) | 29% 가짜 |

**결론: 가짜 PASS 54건 / 전체 약 120건 = ~45%가 실제 검증 없이 통과 처리**

### 조치: 삭제 vs 실제 구현

**삭제해야 할 항목** (이미 유닛/통합 테스트로 커버됨):
- C-2~C-10: 객실 배정 내부 로직 → `test_sms_tag_sync.py`로 커버
- J-2~J-6: 성비 조건 → `test_send_condition.py`로 커버
- N-2~N-5: 변수 치환/버퍼/반올림 → `test_apply_buffers.py`로 커버
- H-6~H-12: 필터 조합 → `test_structural_filters.py`로 커버

**실제 API 호출로 전환해야 할 항목** (E2E에서만 잡을 수 있는 것):
- N-6: SMS/LMS 자동 감지 (90바이트) → 실제 발송 API 호출 후 결과 확인
- N-7: 발송 후 칩 sent_at 업데이트 → run → 칩 조회로 검증
- N-9: ActivityLog 기록 → 발송 후 로그 API 조회로 검증
- L-4~L-6: 칩 보호 규칙 → 수동 칩 생성 후 스케줄 수정 → 칩 잔존 확인
- L-7: 객실 배정 변경 → 칩 동기화 → 배정 API → 칩 API 확인

### 3-3. 예약 CRUD 기본 테스트 (놀랍게도 없음)

기존 E2E에 **예약 CRUD 자체를 테스트하는 섹션이 없음**. Q-10에서 생성만 테스트하고, 수정/삭제는 없음.

| # | 테스트 항목 | 검증 포인트 |
|---|-----------|-----------|
| O-1 | 예약 생성 | POST → 필수 필드 검증, 응답 확인 |
| O-2 | 예약 수정 | PUT → 날짜/인원/상태 변경 반영 |
| O-3 | 예약 삭제 | DELETE → 연관 데이터(칩, 배정) 같이 삭제 |
| O-4 | 예약 목록 조회 + 검색 | GET + search 파라미터 → 이름/전화번호 검색 |
| O-5 | 예약 목록 페이지네이션 | GET + limit/offset → 정확한 건수 |
| O-6 | 예약 상태 변경 | confirmed → cancelled → 칩 정리 확인 |

| 작업 | 예상 효과 |
